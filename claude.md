# الوثيقة الشاملة لمنصة إدارة المشاريع والأسهم الخيرية

**نوع الوثيقة:** التصور العام + مواصفات المتطلبات البرمجية SRS + ملحق الشاشات والحقول ومسارات الاعتماد + المعمارية التقنية  
**الإصدار:** 1.1  
**الحالة:** مسودة موحدة للمراجعة والاعتماد  
**التقنيات المعتمدة:** Django + Django REST Framework + React  
**نموذج النشر:** مستودع واحد، مشروع Django واحد، قالب Django واحد لتقديم React، وصورة Docker واحدة للتطبيق  
**اللغة:** العربية، من اليمين إلى اليسار RTL

---

# 1. الملخص التنفيذي

تهدف المنصة إلى إدارة المشاريع الخيرية وتمكين الأفراد والشركات من المساهمة فيها من خلال أسهم خيرية، مساهمات مفتوحة، اشتراكات دورية، تبرعات مباشرة، ومساهمات عينية.

لا تقتصر المنصة على جمع المساهمات، بل تغطي دورة حياة المشروع كاملة:

- إنشاء المشروع واعتماده ونشره.
- تعريف أنواع الأسهم وقيمها ودوريتها.
- تسجيل المساهمين والاشتراكات والاستحقاقات.
- رفع إثباتات الدفع واعتمادها.
- تسجيل الإيرادات النقدية والعينية.
- تسجيل المصروفات وإرفاق الفواتير.
- إدارة الميزانية ومراحل التنفيذ.
- توفير لوحات معلومات وتقارير.
- إتاحة الشفافية للعامة وفق إعدادات صاحب المشروع.
- إرسال الإشعارات والحملات عبر واتساب.
- إصدار إيصالات رقمية وتقارير ختامية.
- إغلاق المشروع ماليًا وتنفيذيًا.

تعمل المنصة كنظام متعدد الجهات والمشاريع، مع فصل بيانات وصلاحيات كل جهة عن الأخرى.

# 2. الرؤية والأهداف

## 2.1 الرؤية

إنشاء منصة رقمية موثوقة وشفافة وسهلة الاستخدام لتنظيم المشاريع الخيرية، وتعزيز ثقة المجتمع، وتحسين المتابعة المالية والتنفيذية.

## 2.2 الأهداف

1. تسهيل إطلاق المشاريع الخيرية والتعريف بها.
2. تمكين المجتمع من المشاركة من خلال أسهم خيرية واضحة.
3. دعم المساهمات الفردية والاشتراكات الدورية.
4. تمكين المساهم من معرفة المدفوع والمتبقي والمتأخر.
5. توثيق جميع الإيرادات النقدية والعينية.
6. توثيق المصروفات وربطها بالفواتير والمستندات.
7. توفير لوحة مالية وتشغيلية لحظية لكل مشروع.
8. إتاحة مستوى اختياري من الشفافية للعامة.
9. التواصل مع المساهمين عبر واتساب.
10. إصدار تقارير قابلة للطباعة والتصدير.
11. إنشاء سجل تدقيق لجميع العمليات الحساسة.
12. دعم عدة جهات ومشاريع داخل منصة واحدة.

## 2.3 مؤشرات النجاح

- عدد المشاريع المنشورة والمكتملة.
- عدد المساهمين النشطين.
- إجمالي المساهمات.
- نسبة تحصيل الأسهم المسجلة.
- نسبة الدفعات المتأخرة.
- نسبة المصروفات المرفقة بفواتير.
- عدد المشاريع التي فعّلت الشفافية.
- معدل نجاح رسائل واتساب.
- متوسط مدة إكمال المشروع.
- مستوى رضا المساهمين وأصحاب المشاريع.

- إدارة التصنيفات والإعدادات.
- الاطلاع على جميع التقارير.

## 5.2 مدير الجهة

## 5.3 صاحب المشروع

- نشر التحديثات والحملات.
- ضبط الشفافية وطلب الإغلاق.

## 5.4 المسؤول المالي

## 5.5 المدقق

- المشاركة في إغلاق المشروع.

## 5.6 مسؤول المحتوى والحملات

## 5.7 المساهم

# 7. المتطلبات الوظيفية

## 7.1 التسجيل والدخول

- إنشاء حساب برقم الهاتف.
- إرسال OTP عبر واتساب أو قناة بديلة.
- تحديد صلاحية الرمز وعدد المحاولات.
- عدم استخدام الرمز أكثر من مرة.
- تسجيل محاولات الدخول.
- إنهاء جميع الجلسات.
- تحقق إضافي للحسابات الإدارية.

## 7.2 المستخدمون

- عرض الاسم والهاتف والدور والجهة والحالة وآخر دخول.
- تفعيل وتعطيل الحساب.
- تغيير الدور والصلاحيات.
- عدم حذف مستخدم مرتبط بعمليات مالية.
- تسجيل تغييرات الصلاحيات.

## 7.3 الجهات

- إنشاء جهة وتسجيل الاسم والنوع والشعار والوصف والتواصل والمسؤول.
- رفع مستندات التحقق.
- اعتماد أو رفض أو إعادة للتعديل.
- إضافة أعضاء وإسناد أدوار.
- فصل بيانات الجهات.

## 7.4 المشاريع

- إنشاء المشروع كمسودة.
- إدخال الاسم والتصنيف والوصف والجهة والموقع والتواريخ والمستفيدين.
- إدخال التكلفة والهدف والمبلغ المتوفر والعملة ووسائل الدفع.
- حفظ وإرسال للاعتماد.
- عدم النشر قبل الاعتماد.
- اعتماد أو رفض أو إعادة أو تعليق.
- توليد رابط عام وQR.
- إيقاف استقبال المساهمات.
- حفظ تاريخ التغييرات.

## 7.5 الصفحة العامة

- عرض المشاريع النشطة والجديدة والأكثر دعمًا والقريبة من الهدف والمكتملة.
- البحث والتصفية حسب التصنيف والموقع والحالة.
- عرض مؤشرات عامة قابلة للتهيئة.
- عرض بيانات المشروع وفق إعدادات الشفافية.

## 7.6 الأسهم والاشتراكات

- إنشاء أكثر من نوع سهم.
- تحديد القيمة والدورية والحدود.
- دعم مرة واحدة، أسبوعي، شهري، ربع سنوي، نصف سنوي، سنوي، وجدول مخصص.
- دعم مبلغ مفتوح.
- عدم تطبيق تعديل القيمة بأثر رجعي.
- إنشاء رقم اشتراك فريد.
- إنشاء استحقاقات تلقائيًا.
- دعم السداد الجزئي.
- دعم طلب الإيقاف أو الإلغاء وفق السياسة.

## 7.7 المدفوعات

- تسجيل دفعة يدوية.
- تسجيل المبلغ والطريقة والتاريخ والمرجع والإثبات.
- دعم التحويل والإيداع والنقد ونقطة البيع والرابط الخارجي.
- وضع الدفعة بانتظار المراجعة.
- اعتماد أو رفض أو إعادة للاستكمال.
- عند الاعتماد: تحديث الاستحقاقات والاشتراك، إنشاء الإيراد والإيصال، تحديث المؤشرات، وإرسال الإشعار.
- منع تكرار المرجع عند تفعيل السياسة.
- عدم تعديل الدفعة المعتمدة مباشرة.

## 7.8 الإيرادات والمساهمات العينية

- تسجيل إيرادات مرتبطة أو غير مرتبطة بالاشتراكات.
- دعم الأسهم والتبرعات والمنح والشركات والفعاليات.
- دعم مساهم غير مسجل.
- عدم حذف الإيراد المعتمد.
- تسجيل العيني: المادة، الكمية، الوحدة، القيمة، المساهم، الاستلام، الصور والمحضر.
- الفصل بين النقدي والعيني في التقارير.
- جعل احتساب العيني ضمن الإنجاز اختياريًا.

## 7.9 المصروفات

- تسجيل المشروع والتصنيف والمورد والفاتورة والتاريخ والمبلغ والضريبة والطريقة والبند والوصف.
- إلزام الفاتورة حسب حد مالي.
- اعتماد متعدد المراحل.
- عدم حذف المصروف المعتمد.
- التنبيه عند تجاوز الميزانية.
- تحديد ما إذا كان الخصم من الرصيد عند الاعتماد أو الدفع.

## 7.10 الموردون والميزانية

- إدارة بيانات الموردين وحساباتهم ومستنداتهم.
- إنشاء ميزانية وبنود.
- عرض المعتمد والمصروف والالتزامات والمتبقي.
- تنبيهات الاقتراب من التجاوز والتجاوز.
- تسجيل سبب تعديل الميزانية المعتمدة.

## 7.11 الإنجاز التنفيذي

- إنشاء مراحل بأوزان مجموعها 100%.
- تحديد المسؤول والتواريخ والنسبة والحالة.
- حساب الإنجاز تلقائيًا.
- نشر تحديثات عامة أو داخلية مع الصور.

## 7.12 الشفافية

- التحكم في عرض الهدف والمحصل والمتبقي والإيرادات والمصروفات والرصيد والفواتير وأسماء المساهمين والمراحل.
- إخفاء البيانات الحساسة.
- تسجيل تغييرات الشفافية.
- إظهار تاريخ آخر تحديث.

## 7.13 واتساب والحملات

- إرسال نص وقالب وصورة وملف ورابط.
- استقبال حالة الرسالة والفشل.
- حفظ سجل الرسائل وإعادة المحاولة.
- احترام إيقاف الرسائل الترويجية.
- إنشاء حملات لمساهمي المشروع.
- تحديد الجمهور والقالب والموعد.
- معاينة وإرسال اختبار وتأكيد الإرسال الجماعي.

## 7.14 الإيصالات والتقارير

- إصدار إيصال PDF عند اعتماد الدفعة.
- تضمين رقم الإيصال والمشروع والمبلغ والتاريخ وQR.
- صفحة تحقق عامة.
- تقارير المشاريع والإيرادات والمصروفات والأرصدة والاشتراكات والمتأخرات والميزانية والإنجاز.
- تصدير PDF وExcel.

## 7.15 التدقيق والإغلاق

- تسجيل المستخدم والعملية والبيانات القديمة والجديدة والوقت وعنوان IP.
- عدم حذف سجل التدقيق.
- طلب إغلاق المشروع.
- التحقق من العمليات المعلقة والتقرير والرصيد ومعالجة الفائض.
- منع العمليات المالية بعد الإغلاق.

# 9. مسارات الاعتماد

## 9.1 اعتماد الجهة

مسودة → بانتظار التحقق → معتمدة أو معادة للتعديل أو مرفوضة أو موقوفة.

**متطلبات الاعتماد:** اكتمال البيانات، وجود مسؤول، وسيلة تواصل، المستندات، وعدم التكرار.

## 9.2 اعتماد المشروع

مسودة → بانتظار الاعتماد → معتمد → نشط.

ويمكن الانتقال إلى معاد للتعديل أو مرفوض.

**قائمة التحقق:** الوصف، الجهة، الهدف، الصور، وسائل الدفع، الشفافية، أنواع المساهمة، التواريخ، المدير، البيانات البنكية، والتصاريح.

**الحقول الحساسة بعد الاعتماد:** الهدف المالي، الحساب البنكي، الجهة، المدير، تاريخ النهاية، معالجة الفائض، والشفافية المالية؛ وتحتاج طلب تعديل.

## 9.3 اعتماد الدفعة

مسودة → بانتظار المراجعة → معتمدة أو معادة للاستكمال أو مرفوضة → معكوسة عند التصحيح.

عند الاعتماد ينفذ النظام داخل معاملة واحدة:

1. تحديث الدفعة.
2. توزيع المبلغ على الاستحقاقات.
3. تحديث الاشتراك.
4. إنشاء الإيراد.
5. تحديث مؤشرات المشروع.
6. إصدار الإيصال.
7. تسجيل التدقيق.
8. إضافة رسالة واتساب إلى Outbox.

## 9.4 اعتماد المصروف

**منخفض القيمة:** مسودة → اعتماد صاحب المشروع → معتمد → مدفوع.

**متوسط القيمة:** مسودة → مراجعة المسؤول المالي → اعتماد صاحب المشروع → معتمد → مدفوع.

**مرتفع القيمة:** مسودة → مسؤول مالي → مدقق → مدير الجهة أو صاحب المشروع → مدير المنصة عند الحاجة → معتمد → مدفوع.

لا يعتمد منشئ المصروف العملية منفردًا إذا كانت سياسة الفصل بين المهام مفعلة.

## 9.5 اعتماد الميزانية

مسودة → بانتظار الاعتماد → معتمدة أو معادة للتعديل.

تعديل المعتمدة يتطلب السبب والقيم السابقة والجديدة والبنود المتأثرة والأثر والمرفقات والمعتمد.

## 9.6 المساهمة العينية

مسجلة → بانتظار التحقق → مقبولة → مستلمة → مستخدمة جزئيًا أو بالكامل، أو مرفوضة أو ملغاة.

## 9.7 تحديث المشروع

مسودة → بانتظار المراجعة → مجدول → منشور → مؤرشف.

## 9.8 حملة واتساب

مسودة → بانتظار الاعتماد → معتمدة → مجدولة → قيد الإرسال → مكتملة أو جزئية أو فاشلة أو ملغاة.

## 9.9 إغلاق المشروع

نشط أو قيد التنفيذ → طلب إغلاق → مراجعة مالية → مراجعة تنفيذية → مكتمل → مغلق ماليًا.

**موانع الإغلاق:** دفعات أو مصروفات معلقة، رصيد غير مفسر، ميزانية غير معتمدة، تقرير ناقص، فائض غير محدد، استحقاقات مستقبلية أو حملات مجدولة.

# 10. قواعد العمل

1. لا ينشر المشروع قبل الاعتماد.
2. لا تعتبر المساهمة مدفوعة قبل اعتماد الدفعة.
3. كل دفعة معتمدة تنتج إيرادًا وإيصالًا.
4. لا تحذف العمليات المالية المعتمدة.
5. التصحيح من خلال عكس أو سجل جديد.
6. لا يتجاوز توزيع الدفعة قيمتها.
7. مجموع أوزان المراحل 100%.
8. الصلاحيات تطبق في الخادم وليس الواجهة فقط.
9. لا يظهر اسم المساهم دون موافقته.
10. لا تظهر البيانات الحساسة في الصفحات العامة.
11. لكل اشتراك ودفعة وإيراد ومصروف وإيصال رقم فريد.
12. تعطيل المستخدم لا يحذف عملياته.
13. تعديل عملية معتمدة يتطلب سببًا.
14. حد الموافقات الإضافية قابل للتهيئة.
15. احتساب العيني اختياري.
16. تحديث المؤشرات بعد اعتماد العمليات.
17. لا ترسل الحملات لمن أوقفها.
18. تحفظ حالات واتساب.
19. إلغاء الاشتراك يوقف الاستحقاقات المستقبلية فقط.
20. لا تعدل قيمة السهم بأثر رجعي.
21. تحدد معالجة الفائض قبل الإغلاق.
22. يمكن منع المصروف المتجاوز للرصيد.
23. يظهر تاريخ آخر تحديث للشفافية.
24. لا يراسل صاحب مشروع مساهمي مشروع آخر.
25. تستخدم Decimal لجميع القيم المالية.
26. تحفظ التواريخ بتوقيت موحد.
27. تحفظ أرقام الهاتف بالتنسيق الدولي.

# 11. حالات الاستخدام الرئيسية

1. **إنشاء حساب:** هاتف → OTP → بيانات → موافقات → حساب فعال.
2. **إنشاء مشروع:** بيانات → هدف → أسهم → مستندات → شفافية → إرسال.
3. **اعتماد مشروع:** مراجعة → اعتماد أو إعادة أو رفض → إشعار.
4. **تسجيل سهم:** مشروع → سهم → عدد → دورية → تحقق → اشتراك واستحقاقات.
5. **رفع إثبات:** استحقاق → مبلغ ومرجع → ملف → إرسال للمراجعة.
6. **اعتماد دفعة:** مراجعة → اعتماد → إيراد وإيصال ومؤشرات وإشعار.
7. **تسجيل مصروف:** بيانات وفاتورة وميزانية → اعتماد.
8. **نشر تحديث:** محتوى وصور ومرحلة → مراجعة → نشر.
9. **حملة واتساب:** جمهور وقالب → معاينة واختبار → اعتماد وإرسال.
10. **إغلاق مشروع:** طلب → فحص ومراجعة → تقرير وفائض → إغلاق.

# 14. تكامل واتساب

## 14.1 الاستخدامات

- OTP.
- تأكيد التسجيل والاشتراك.
- تذكير الاستحقاق.
- تأكيد أو رفض الدفع.
- إرسال الإيصال.
- تحديثات المشروع.
- الحملات والرسالة الختامية.

## 14.2 متطلبات التكامل

- عنوان API وآلية المصادقة.
- إرسال النص والقوالب والصور والملفات والروابط.
- استقبال حالة الرسالة أو Webhook.
- حدود الإرسال ومعالجة الأخطاء.
- إعادة المحاولة ومنع التكرار.

## 14.3 نمط Outbox

1. تسجل الرسالة في جدول Outbox بحالة pending.
2. عامل داخلي يقرأ الرسائل.
3. يرسل إلى بوابة واتساب.
4. يحدث الحالة والمرجع الخارجي.
5. يعيد المحاولة وفق السياسة.
6. لا يؤدي فشل واتساب إلى فشل العملية المالية.

# 18. تشغيل المشروع في Docker واحد

## 18.1 الإمكانية

نعم، يمكن إنشاء صورة Docker واحدة وحاوية تطبيق واحدة تضم:

- Django.
- React بعد البناء.
- Gunicorn.
- WhiteNoise للملفات الثابتة.
- عامل Outbox داخلي عند الحاجة.
- اتصال بقاعدة بيانات خارجية أو SQLite داخل Volume.

## 18.2 الخيار الموصى به للإنتاج

**حاوية تطبيق واحدة + PostgreSQL خارجية**:

- Docker واحد للتطبيق.
- PostgreSQL مُدارة أو على خادم منفصل.
- Volume أو Object Storage للمرفقات.
- Gunicorn وWhiteNoise.

هذا يحافظ على بساطة النشر ولا يضع قاعدة البيانات داخل حاوية التطبيق.

## 18.3 النسخة التجريبية

يمكن استخدام SQLite داخل Volume، وهو مناسب للتجربة أو الاستخدام المحدود، لكنه غير مناسب لعدد كبير من العمليات المالية المتزامنة.

## 18.4 PostgreSQL داخل الحاوية نفسها

ممكن تقنيًا لكنه غير موصى به بسبب صعوبة النسخ الاحتياطي والتوسع وخلط مسؤوليات التطبيق والبيانات.

## 18.5 Docker Multi-stage Build

```dockerfile
FROM node:alpine AS frontend-build
WORKDIR /frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

FROM python:slim AS runtime
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
COPY --from=frontend-build /frontend/dist /app/static/frontend
RUN python manage.py collectstatic --noinput
CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

## 18.6 بدء الحاوية

يفضل `entrypoint.sh` لتنفيذ:

1. فحص قاعدة البيانات.
2. migrations.
3. collectstatic.
4. تهيئة المجلدات.
5. تشغيل Gunicorn.
6. تشغيل عامل الإشعارات إذا كان داخل الحاوية.

## 18.7 المهام المجدولة

للحفاظ على حاوية واحدة:

- تشغيل Gunicorn وعامل Management Command باستخدام مشرف عمليات، أو
- استخدام Scheduled Jobs من منصة الاستضافة لتشغيل:

```text
python manage.py process_outbox
python manage.py send_due_reminders
```

## 18.8 الصحة والمراقبة

```text
/health/live/
/health/ready/
```

- Live: التطبيق يعمل.
- Ready: قاعدة البيانات والتهيئة جاهزة.
